# 策划战斗调试工具 v2 设计

## 目的

在现有最小战斗调试场景基础上，做一个更适合策划使用的测试工具。它不是正式战斗 UI，而是一个能快速验证规则、数值和回合流转的内部调试场景。

本阶段固定使用当前测试战斗内容：

- 玩家：现有测试玩家
- 卡牌：`Strike`、`Defend`
- 敌人：`DummyEnemy`

目标是让策划不用改代码，也能观察并调整当前战斗状态，验证打牌、结束回合、格挡抵消、牌堆流转和日志输出是否符合预期。

## 当前基础

项目已经具备：

- `scripts/stm/` 规则骨架
- `ActionQueue` 行动队列
- `Combat` 战斗流程
- `Player`、`CardManager`、`Enemy`、`Creature`
- 测试用 `Strike`、`Defend`、`DummyEnemy`
- 最小调试场景 `scenes/stm/battle_debug_scene.tscn`
- 场景脚本 `scripts/stm/debug/battle_debug_scene.gd`
- GUT 测试 `scripts/stm/tests/test_battle_debug_scene.gd`

本设计是在现有调试场景上增强，不重写规则层，不新增正式卡牌库或敌人库。

## 用户定位

主要用户是策划。使用者应能通过界面完成以下测试：

- 查看玩家血量、能量、格挡
- 查看敌人血量、敌人意图和攻击数值
- 查看手牌、抽牌堆、弃牌堆
- 点击 `Strike`、`Defend`、`结束回合`
- 点击 `重开战斗`
- 输入并应用基础数值
- 阅读简洁日志，必要时打开详细日志

工具允许暴露规则层细节，但默认显示应偏策划可读，而不是程序调试日志。

## 非目标

本阶段不做以下内容：

- 不做正式战斗 UI
- 不做正式卡牌选择器
- 不做正式敌人选择器
- 不做卡牌编辑器
- 不做完整控制台
- 不做保存/读取调试配置
- 不做多敌人战斗
- 不接入 `powers`、遗物、药水、房间、地图或奖励流程
- 不修改 Python 参考项目 `slay-the-model-main/`

## 界面结构

调试场景保持单屏工具布局，优先清晰和稳定。

建议分区如下：

1. 标题区
   - 显示工具名称，例如 `战斗调试工具`
   - 可显示当前战斗状态，例如 `进行中`、`战斗胜利`、`战斗失败`

2. 玩家状态区
   - 玩家血量：`当前 HP / 最大 HP`
   - 能量：`当前能量 / 最大能量`
   - 格挡

3. 敌人状态区
   - 敌人名称
   - 敌人血量：`当前 HP / 最大 HP`
   - 敌人意图
   - 预计攻击数值

4. 牌堆区
   - 手牌列表
   - 抽牌堆列表或数量加摘要
   - 弃牌堆列表或数量加摘要

5. 操作区
   - `Strike`
   - `Defend`
   - `结束回合`
   - `重开战斗`

6. 数值编辑区
   - 玩家血量输入框
   - 玩家能量输入框
   - 玩家格挡输入框
   - 敌人血量输入框
   - `应用数值` 按钮

7. 日志区
   - 简洁日志显示
   - `显示详细日志` 勾选项
   - 详细日志开启后，在同一区域追加或替换显示更细的规则过程

## 操作行为

### 打出 Strike

点击 `Strike` 后：

- 查找手牌中的第一张 `Strike`
- 如果可以打出，则通过 `Combat.play_card()` 执行
- 刷新玩家状态、敌人状态、牌堆显示和按钮状态
- 简洁日志写入类似：`打出 Strike，敌人受到 6 点伤害`
- 详细日志开启时，额外记录能量变化、弃牌、行动结算和终局检查

如果没有 `Strike` 或敌人不存在：

- 不改变战斗状态
- 写入错误或提示日志
- 按钮应被禁用或点击后给出明确提示

### 打出 Defend

点击 `Defend` 后：

- 查找手牌中的第一张 `Defend`
- 如果可以打出，则通过 `Combat.play_card()` 执行
- 刷新显示
- 简洁日志写入类似：`打出 Defend，获得 5 点格挡`
- 详细日志开启时，额外记录能量变化、弃牌和行动结算

### 结束回合

点击 `结束回合` 后：

- 通过 `Combat.end_turn()` 执行玩家回合结束和敌人回合
- 进入下一玩家回合
- 刷新玩家血量、能量、格挡、手牌和牌堆
- 简洁日志写入类似：`结束回合，DummyEnemy 攻击，玩家损失 1 点 HP`
- 详细日志开启时，额外记录弃掉手牌、敌人意图、攻击动作、格挡抵消、抽牌和阶段切换

### 重开战斗

点击 `重开战斗` 后：

- 重新创建固定测试战斗
- 清空当前日志并写入一条新战斗开始日志
- 重置输入框为当前初始状态
- 刷新所有显示和按钮状态

### 应用数值

点击 `应用数值` 后，读取输入框并应用：

- 玩家血量
- 玩家能量
- 玩家格挡
- 敌人血量

应用规则：

- 输入必须是整数
- 血量不能低于 0
- 玩家血量不能高于玩家最大血量
- 敌人血量不能高于敌人最大血量
- 能量不能低于 0
- 格挡不能低于 0

非法输入时：

- 不应用任何数值变更
- 保留当前战斗状态
- 写入简洁错误日志，例如 `数值应用失败：玩家血量必须是整数`
- 详细日志开启时可列出具体字段和原始输入

为避免半成功状态，`应用数值` 应先完成所有输入校验，再一次性写入状态。

## 日志设计

日志分为两层。

### 简洁日志

默认显示。面向策划阅读，只描述关键结果：

- `战斗开始`
- `打出 Strike，敌人受到 6 点伤害`
- `打出 Defend，获得 5 点格挡`
- `结束回合，敌人攻击造成 1 点伤害`
- `应用数值：玩家 HP 设为 40，敌人 HP 设为 10`
- `战斗胜利`
- `输入错误：敌人血量必须是整数`

### 详细日志

通过 `显示详细日志` 开关显示。面向规则排查，记录更细的过程：

- 抽牌动作
- 卡牌进入弃牌堆
- 能量消耗
- 敌人意图执行
- 格挡吸收伤害
- 阶段切换
- 行动队列返回的终局结果

详细日志不要求完整覆盖所有未来规则，但本阶段应覆盖 `Strike`、`Defend`、结束回合、抽牌和数值应用。

## 数据流

调试场景只负责显示和调用规则层，不拥有核心规则。

基础流转：

1. `StmBattleDebugScene` 创建测试战斗
2. 场景读取 `GameState`、`Player`、`Combat`、`Enemy`、`CardManager`
3. 用户点击按钮
4. 场景调用规则层公开方法
5. 规则层修改状态
6. 场景刷新所有标签、输入框、按钮和日志

数值编辑是调试工具特权，但仍应通过明确方法集中处理，不应把散落的状态修改写在多个按钮回调里。

## 错误处理

应处理以下情况：

- 找不到 `Strike`
- 找不到 `Defend`
- 没有存活敌人
- 战斗对象为空
- 玩家对象为空
- 输入框为空
- 输入不是整数
- 输入小于 0
- 输入超过最大血量

错误处理原则：

- 不崩溃
- 不产生半应用状态
- 写入简洁日志
- 详细日志开启时给出更多上下文

## 测试策略

继续使用 GUT，并遵守 `AGENT.md` 的 BDD 规则。

新增或修改测试前，必须先在测试方法中写中文 Given-When-Then 行为注释，再写测试代码，再写正式代码。

建议测试覆盖：

1. 初始界面
   - 显示玩家血量、能量、格挡
   - 显示敌人血量、意图、攻击数值
   - 显示手牌、抽牌堆、弃牌堆
   - 显示数值输入框、应用按钮、重开按钮、详细日志开关

2. 应用合法数值
   - 修改玩家血量
   - 修改玩家能量
   - 修改玩家格挡
   - 修改敌人血量
   - 刷新界面
   - 写入简洁日志

3. 应用非法数值
   - 非数字
   - 空值
   - 负数
   - 超过最大血量
   - 当前状态不被修改
   - 写入错误日志

4. 打牌日志
   - 点击 `Strike` 后状态刷新并写入日志
   - 点击 `Defend` 后状态刷新并写入日志

5. 结束回合日志
   - 点击 `结束回合` 后敌人攻击、格挡抵消、下一回合抽牌
   - 简洁日志包含关键结果

6. 详细日志开关
   - 默认显示简洁日志
   - 开启后显示或追加详细日志
   - 关闭后回到简洁日志

7. 重开战斗
   - 状态恢复初始战斗
   - 日志重置
   - 输入框恢复为当前状态
