## 0. 最高优先级修正：酒馆战棋始终是第一参照

本项目的核心玩法目标来自自走棋，尤其是炉石传说酒馆战棋。

因此，Daggerheart 的 Hope / Fear 不能取代酒馆战棋作为主要类比源。

必须避免以下误读：

```text
错误 1：把 Hope / Fear 理解成抽象叙事余波轴。
错误 2：把 Hope / Fear 僵硬落成四个固定结果格。
错误 3：让 TRPG 投骰结果结构覆盖自走棋自动团战过程。
```

正确理解：

```text
Daggerheart 只提醒我们：成功率和最终结果之间，还有“过程里赚了什么、亏了什么”。
真正应参考的是酒馆战棋自动团战：胜率只是观看预期，实际过程里会触发大量收益和损失。
```

---

## 1. 资料摘要

Daggerheart 的行动检定使用两颗不同颜色的 d12，分别代表 Hope 与 Fear。

规则上，它不是只判断“成功 / 失败”，而是同时判断两条轴：

```text
1. 总点数是否达到难度：决定成功 / 失败。
2. Hope die 与 Fear die 谁更高：决定结果带 Hope 还是带 Fear。
```

因此形成四种基本结果：

```text
Success with Hope
Success with Fear
Failure with Hope
Failure with Fear
```

官方 SRD 对四种结果的简写近似为：

```text
Success with Hope = Yes, and...
Success with Fear = Yes, but...
Failure with Hope = No, but...
Failure with Fear = No, and...
```

Daggerheart 还强调 fail forward：每次掷骰都会让场景发生变化，不存在“什么都没发生”的无效检定。

本项目不学习投骰，不学习 Hope/Fear 代币，不照搬四个格子。只学习一个 MDA 启发：

```text
结果不应该只有“成功率开奖”。
实际执行过程中的收益和损失，才是玩家观看结果的重要快感来源。
```

## 2. MDA 拆解

### 2.1 Mechanics：机制层

Daggerheart 的机制层特点是把一次行动结果拆成两个维度：

```text
行动是否达成目标。
行动过程中附带了怎样的正负价值。
```

在 Daggerheart 原规则中，这会被表达成 Success / Failure 与 Hope / Fear 的组合。

但在本项目中，不能把它僵硬实现成四象限。

更适合 Pressure Encounter 的机制表达是：

```text
outcome_rate：dominant_action 达成主目标的概率。
auto_execution_events：自动执行过程中实际触发的收益 / 损失 / 保留 / 成长 / 伤害 / 下一轮压力。
final_consequence：主目标最终是否达成。
value_summary：这一轮自动执行实际赚了什么、亏了什么。
```

### 2.2 Dynamics：动态层

Daggerheart 带来的动态启发是：玩家不只关心“我成功了吗”，还关心：

```text
我成功的过程中有没有赚到额外东西？
我成功的过程中有没有付出代价？
我失败时有没有保住资源或获得成长？
我失败时是不是还让局势进一步恶化？
```

这与酒馆战棋自动团战高度一致。

玩家看自动团战时，也不只看胜负：

```text
赢了能打多少血？
有没有触发复仇？
有没有触发亡语？
核心随从有没有活下来？
输了有没有完成成长？
对面虽然输了，是否通过亡语 / 复仇 / 任务获得了巨大收益？
```

### 2.3 Aesthetics：体验层

该结构产生的体验不是“选择一个象限”，而是：

```text
带着预期看事情发生。
看过程如何验证或打破预期。
看自动执行中触发的收益和损失。
看主目标结果之外，局势如何被重新塑形。
```

这正是本项目需要的自动执行快感。

## 3. 用酒馆战棋重新解释 Daggerheart 启发

酒馆战棋是本项目最重要的参考。

在酒馆战棋里，胜率插件显示的只是预期：

```text
这场团战大概能不能赢。
```

但玩家真正观看的是过程：

```text
有没有触发复仇？
有没有触发亡语？
有没有触发圣盾、剧毒、召唤、成长等连锁？
赢了能打对面多少血？
输了有没有保住关键随从？
输了有没有因为战斗过程获得成长？
对面虽然输了，是否通过亡语 / 复仇 / 任务 / 成长获得了巨大收益？
这场团战有没有改变下一回合的经济与战力格局？
```

因此，Daggerheart 的启发在本项目里应翻译为：

```text
不要把结果只做成主目标成功/失败。
也不要把结果硬分成四格。
要让自动执行过程展示主目标之外的收益和损失。
```

## 4. 对 Pressure Encounter 的正确映射

Pressure Encounter 已有结构：

```text
压力积攒环节
→ dominant_action 已经由行动倾向竞争决出
→ outcome_rate 显示该 dominant_action 的主目标成功率
→ 自动执行环节展示事情如何不可避免地发生
```

因此 Daggerheart 的参考点应该落在自动执行过程，而不是投骰或四象限标签。

正确映射：

```text
成功率轴：dominant_action 是否达成主目标。
过程价值事件：自动执行过程中触发了哪些收益、损失、保留、成长、伤害和下一轮压力。
```

自动执行中的收益/损失可以包括：

```text
关系协同触发。
情绪污染打断。
证据降低误判代价。
保留的关键念头发挥作用。
强硬行动压住局面但损害信任。
失败暴露真正矛盾，打开下一轮机会。
压力残留进入下一轮。
同伴获得行动窗口。
角色承受伤害。
局势虽然被压住，但谈判空间减少。
```

这些事件比一个硬四象限标签更重要。

## 5. 用《极乐迪斯科》枪战理解

玩家在枪战前的压力积攒相当于酒馆操作：

```text
对白选择。
白色检定。
装备准备。
关系积累。
信息理解。
恐惧处理。
临场念头筛选。
```

进入枪战时，系统已经决定 dominant_action，例如：

```text
dominant_action：强硬干预
主目标：阻止枪战立刻全面失控
outcome_rate：62%
```

但自动执行不是：

```text
成功：阻止失控。
失败：没阻止失控。
```

更像酒馆团战的是过程：

```text
你抢先开口，压住对方半秒。
Kim 抓住窗口行动。
你之前没有处理好的恐惧让你迟疑了一拍。
同伴信任不足，没有完全跟上。
对方仍然造成伤害。
你保住了一个关键判断，但牺牲了谈判余地。
局势没有彻底崩，但下一轮压力更高。
```

玩家带着 62% 成果率观看这个过程，快感来自：

```text
我知道主行动大概能不能成。
但我不知道这场“压力团战”里会触发哪些收益和损失。
我会看着每个积累过的念头、关系、污染、核心触发在过程里发挥作用或造成代价。
```

## 6. 对 v1.1 的建议

v1.1 不需要做复杂 Hope / Fear 资源系统，也不需要硬编码四象限结果表。

建议概念：

```text
outcome_rate：dominant_action 的主目标成功率。
auto_execution_events：自动执行过程事件流。
final_consequence：主目标是否达成，以及局势进入什么状态。
value_summary：这一轮执行实际赚了什么、亏了什么。
```

v1.1 可以先支持几类自动执行事件：

```text
主目标推进事件：行动是否接近成功。
关系协同事件：同伴是否跟上、信任是否吸收代价。
情绪干扰事件：恐惧、迟疑、僵住是否打断执行。
关键件收益事件：保留 / 表达的念头是否在执行中触发价值。
代价事件：伤害、压力残留、关系损伤、下一轮风险。
下一轮铺垫事件：失败但获得信息，或成功但留下新压力。
```

v1.1 最小可先实现 4 类：

```text
主目标推进
关系协同
情绪干扰
代价 / 铺垫
```

## 7. 暂不做内容

v1.1 暂不做：

```text
Hope / Fear 代币系统。
GM Fear 资源。
完整骰子模拟。
复杂四象限概率预测。
多行动分布预测。
复杂胜率模拟器。
硬四象限结果表。
```

只做：

```text
把自动执行结果从“成功率开奖”，升级为“带着成功率观看收益/损失事件流”。
```
